**主题**: 分包结构与 pi-mono 对齐  

---

## What to do

确定 py-mono 的分包结构，与 pi-mono 保持架构对齐。

**pi-mono 结构**:
```
packages/
├── ai/              # AI 核心（Provider、流式协议）
├── agent/           # Agent 运行时（Agent 类、Loop、类型定义）
├── coding-agent/    # CLI 应用 + 内置工具实现
│   └── src/core/tools/   # bash, read, write, edit...
├── mom/             # 记忆系统
├── pods/            # 进程管理
├── tui/             # 终端 UI
└── web-ui/          # Web 界面
```

**关键发现**: Agent 包只有类型定义（`AgentTool` 接口），工具实现在 coding-agent 中。

---

## How

### py-mono 当前状态
- ✅ `packages/ai/` - 已完成（Kimi Provider、StreamWatcher）
- ⏳ `packages/agent/` - 待实现（Agent 类、Loop、types）
- ❓ 工具实现 - 需要决策

### 工具实现方案

**方案 A**: 复刻 `packages/coding-agent/`
- 优点: 与 pi-mono 完全一致
- 缺点: 需要完整 CLI/TUI，工作量大

**方案 B**: `packages/agent-tools/` 或 `packages/tools/`
- 优点: 轻量，专注工具实现
- 缺点: 与 pi-mono 不完全一致

**方案 C**: `packages/agent/tools/`
- 优点: 简单直接
- 缺点: 破坏分层，agent 包混入实现

---

## Why

1. **架构对齐**: pi-mono 明确区分 agent（运行时）和 coding-agent（应用）
2. **关注点分离**: agent 负责编排，tools 负责执行
3. **当前需求**: py-mono 只需要工具实现，不需要完整 CLI

**推荐**: 先实现 `packages/agent/`，工具暂放 `packages/agent/tools/`，后续视需求拆分。

---

## Key Decisions

- Agent 包只包含运行时（与 pi-mono 一致）
- 工具实现暂放在 agent/tools/（简化起步）
- 不创建完整的 coding-agent 包（当前不需要 CLI）
