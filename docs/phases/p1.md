# 我的分析和设计方案

## 1. 参考项目架构对比

### pi-mono：事件驱动流式架构（优先参考）

**核心设计：**
- **事件流**：`AssistantMessageEventStream` 基于事件推送（text_start/text_delta/text_end/toolcall_start 等）
- **注册表模式**：`registerApiProvider()` 注册 provider，支持动态扩展
- **双 API 模式**：`stream()`（原始完整控制）+ `streamSimple()`（简化使用）
- **API 与 Provider 分离**：API 是协议类型（anthropic-messages），Provider 是服务商（anthropic/github-copilot）
- **精细化 Usage**：input/output/cacheRead/cacheWrite/cost 分层追踪

**代码组织：**
```
src/
├── types.ts              # 核心类型（Message, Tool, Usage, AssistantMessageEvent）
├── api-registry.ts       # Provider 注册表
├── providers/
│   ├── anthropic.ts      # Anthropic 实现（streamAnthropic, streamSimpleAnthropic）
│   └── register-builtins.ts  # 注册所有内置 provider
└── utils/
    └── event-stream.ts   # AssistantMessageEventStream 实现
```

**关键优势：**
- ✅ 事件流模式 = 消费端无感知底层差异，易于测试和拦截
- ✅ 注册表 = 新 provider 无需修改核心代码
- ✅ 双 API = 简单场景用 streamSimple，复杂场景用 stream
- ✅ Usage 成本追踪精确到 cache read/write

### kimi-cli：协议抽象优先

**核心设计：**
- `ChatProvider` Protocol 定义接口
- `StreamedMessage` Protocol 抽象流式消费
- 自动生成 cache_control，支持 thinking effort 链式配置

**局限：**
- 缺少统一的事件系统
- 注册机制简单（硬编码 match-case）

### opencode：SDK 封装

**核心设计：**
- 依赖 Vercel AI SDK 统一封装
- 配置验证使用 zod

**局限：**
- 无法直接控制底层特性（cache control, thinking）

## 2. 设计决策：优先参考 pi-mono

### 结论：**采用 pi-mono 的事件驱动 + 注册表架构**

**为什么不选 kimi-cli？**
Protocol 抽象虽好，但缺少事件系统，流式数据只能通过 AsyncIterator 推送，消费端难以做精细控制（如暂停、重试、转换）。

**为什么选 pi-mono？**
1. 事件流模式 = 生产级可观测性，每个数据块都有生命周期事件
2. 注册表 = 真正的插件化，无需修改 core 添加 provider
3. 双 API 设计 = 兼顾易用性和灵活性
4. Usage 成本追踪 = 精确到 cache，生产环境必需

### 3.3 事件流系统（核心）

```python
class MessageEventStream:
    """
    事件流核心类，消费端通过 async for 接收事件。
    每个数据块都有 start -> delta* -> end 的生命周期。
    """

    def __init__(self):
        self._queue: asyncio.Queue[MessageEvent | None] = asyncio.Queue()
        self._done = False

    def push(self, event: MessageEvent) -> None:
        """生产端推送事件"""
        if self._done:
            raise RuntimeError("Stream 已结束")
        self._queue.put_nowait(event)

    def end(self, message: AssistantMessage | None = None) -> None:
        """结束流"""
        self._done = True
        self._queue.put_nowait(None)

    def __aiter__(self) -> AsyncIterator[MessageEvent]:
        return self

    async def __anext__(self) -> MessageEvent:
        if self._done and self._queue.empty():
            raise StopAsyncIteration
        item = await self._queue.get()
        if item is None:
            raise StopAsyncIteration
        return item
```

### 3.4 Provider 注册表

```python
# registry.py
_registry: dict[str, ProviderEntry] = {}

class ProviderEntry(BaseModel):
    api: str                                    # API 类型，如 "anthropic-messages"
    stream: Callable[..., MessageEventStream]   # 原始流函数
    stream_simple: Callable[..., MessageEventStream]  # 简化流函数

def register_provider(entry: ProviderEntry) -> None:
    """注册 Provider"""
    _registry[entry.api] = entry

def get_provider(api: str) -> ProviderEntry | None:
    """获取 Provider"""
    return _registry.get(api)

def create_stream(
    api: str,
    model: Model,
    context: Context,
    options: StreamOptions | None = None
) -> MessageEventStream:
    """工厂函数，创建指定 API 的流"""
    provider = get_provider(api)
    if not provider:
        raise ValueError(f"未知的 API: {api}")
    return provider.stream(model, context, options)
```

# errors.py

class AIError(Exception):
    """AI 模块基础错误"""
    pass

class ProviderError(AIError):
    """Provider 错误"""
    def __init__(self, message: str, api: str | None = None):
        super().__init__(message)
        self.api = api

class APIStatusError(ProviderError):
    """API 返回 4xx/5xx"""

        super().__init__(message, api)

## 4. 关键设计决策

| 问题 | 决策 | 理由 |
|------|------|------|
| 流式抽象？ | 事件流（MessageEventStream） | pi-mono 设计，精细化控制每个数据块生命周期 |
| 如何扩展 Provider？ | 注册表模式 | 无需修改 core，import 即注册 |
| 双 API 模式？ | 保留 stream + stream_simple | 简单场景简化配置，复杂场景完全控制 |
| Usage 追踪？ | input/output/cache_read/cache_write/cost | 精确成本核算 |
| 异常处理？ | 细分错误类型 | 便于调用方做重试/降级策略 |
| 消息格式？ | 类 OpenAI 格式 | 业界标准，易于理解 |
