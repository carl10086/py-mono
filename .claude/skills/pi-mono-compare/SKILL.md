# Pi-Mono 对比分析器

用于对比 py-mono (Python) 和 pi-mono (TypeScript) 项目模块的功能实现差异，帮助识别功能缺失和架构对齐问题。

## 工作流程

### 1. 读取目标模块代码

当用户提供模块名称时，按顺序读取以下位置的代码：

**py-mono (当前项目)**：
- `packages/{module}/src/` - 主要源代码
- `packages/{module}/src/{module}/` - 子模块（如果存在）
- `packages/{module}/README.md` - 文档（如果存在）

**pi-mono (参考项目)**：
- `refer/pi-mono/packages/{module}/src/` - TypeScript 源代码
- `refer/pi-mono/packages/{module}/README.md` - 文档

### 2. 分析维度

对每个模块从以下维度进行对比：

#### A. 文件结构对比
- 列出两边的文件清单
- 识别缺失的文件
- 对比目录结构差异

#### B. 核心类型/接口对比
- Python 类型 vs TypeScript 接口
- 字段名称和类型映射
- 可选/必填字段差异

#### C. 核心功能对比
- 类和方法的存在性
- 方法签名对比
- 功能实现完整性

#### D. 架构模式对比
- 设计模式一致性
- 调用流程对比
- 事件/生命周期管理

### 3. 生成对比报告

使用以下格式输出：

```markdown

| 功能点 | pi-mono | py-mono | 状态 |
|--------|---------|---------|------|
| {功能1} | ✅ | ✅ | 完整 |
| {功能2} | ✅ | ❌ | 缺失 |
| {功能3} | ✅ | ⚠️ | 部分 |

## 🔴 关键缺失（阻断功能）

### 1. {缺失功能名称}
**pi-mono 实现**:
```typescript
// 代码片段或描述
```

**py-mono 现状**:
- 状态：完全缺失 / 部分实现
- 影响：{说明此功能缺失的后果}

**建议优先级**：P0 / P1 / P2

## 🟡 次要差异

1. **{差异点}**
   - pi-mono: {描述}
   - py-mono: {描述}
   - 影响：{说明}

## 💡 建议的优先级

| 优先级 | 任务 | 工作量 | 影响 |
|--------|------|--------|------|
| P0 | {任务} | {大小} | {说明} |
| P1 | {任务} | {大小} | {说明} |

## 我的问题

1. 是否优先实现 **{最高优先级任务}**？
2. packages/ai 模块是否已可用？（Agent 等模块依赖 AI 的流式接口）
3. 是否有特定的截止日期或里程碑需要考虑？
```

### 4. 对比原则

- **客观描述**：只描述实际观察到的差异，不做主观价值判断
- **代码引用**：关键差异引用具体代码行号或函数名
- **影响分析**：说明每个缺失/差异对整体功能的影响
- **优先级建议**：根据功能重要性给出 P0/P1/P2 分级

### 5. 特殊处理

**当模块不存在时**：
- 检查是否是拼写错误（agent vs agents）
- 确认 pi-mono 中是否有同名或类似模块
- 说明该模块是否只在一边存在

**当代码量大时**：
- 优先读取核心文件（agent.ts, types.ts, index.ts 等）
- 关注公共 API 而非内部实现细节
- 必要时使用 grep 搜索关键函数

**类型系统差异**：
- Python: 使用 type hints（`str | None`, `list[T]` 等）
- TypeScript: 使用 interface/type
- 忽略语法差异，关注语义等价性

## 输出示例

用户输入："对比 agent 模块"

输出结构：
1. 文件结构对比
2. 核心功能表格
3. 详细差异分析（关键缺失、次要差异）
4. 优先级建议
5. 待确认问题

## 注意事项

- 保持报告简洁，聚焦关键差异
- 使用表格提高可读性
- 用 🔴🟡🟢 等 emoji 标注严重程度
- 最后主动提出澄清问题
- 如果 pi-mono 参考目录不存在，告知用户
